    import { Pipe, PipeTransform } from '@angular/core';

    @Pipe({
    name: 'reverse',
    standalone: true // Set to true if you are using Standalone Components
    })
    export class ReversePipe implements PipeTransform {

    transform(value: string): string {
        if (!value) return '';
        
        // 1. Split into array: "abc" -> ["a", "b", "c"]
        // 2. Reverse array: ["c", "b", "a"]
        // 3. Join back: "cba"
        return value.split('').reverse().join('');
    }

    }

Pipe More than One value
    import { Pipe, PipeTransform } from '@angular/core';

    @Pipe({
    name: 'reverse',
    standalone: true
    })
    export class ReversePipe implements PipeTransform {

    // 'value' is the string to reverse
    // 'isUpper' and 'suffix' are our two custom parameters
    transform(value: string, isUpper: boolean = false, suffix: string = ''): string {
        if (!value) return '';

        let reversed = value.split('').reverse().join('');

        if (isUpper) {
        reversed = reversed.toUpperCase();
        }

        return reversed + suffix;
    }
    }

    import { Pipe, PipeTransform } from '@angular/core';

    @Pipe({
        name: 'calculateTax',
        standalone: true // Use standalone for modern Angular (2026)
    })
    export class CalculateTaxPipe implements PipeTransform {

        // val: The main data (e.g., price)
        // taxRate: The first input
        // currencySymbol: The second input
        transform(val: number, taxRate: number, currencySymbol: string): string {
            if (!val) return '';
            
            const total = val + (val * (taxRate / 100));
            return `${currencySymbol}${total.toFixed(2)}`;
        }
    }

    In your world of development (specifically Angular and Web Dev), it’s the most common 
    way to display data from your TypeScript code onto your HTML screen.

    Interpolation allows you to incorporate dynamic text into your HTML templates. It uses the "double curly brace" syntax: {{ }}.

Event Loop
    The Event Loop is the secret sauce that allows Node.js and 
    JavaScript to be "non-blocking" and handle thousands of concurrent connections, 
    even though JavaScript is single-threaded
    1. The Priority Rule: Micro vs. Macro
        Code: 
            console.log("Step 1: Synchronous");

            setTimeout(() => {
            console.log("Step 2: Macrotask (Timeout)");
            }, 0);

            Promise.resolve().then(() => {
            console.log("Step 3: Microtask (Promise 1)");
            }).then(() => {
            console.log("Step 4: Microtask (Promise 2) - Still Priority!");
            });

            console.log("Step 5: Synchronous End");

        Output: 1 → 5 → 3 → 4 → 2

    2. Async/Await: Just Syntactic Sugar
        Code:
            async function eventLoopTest() {
            console.log("A: Inside Async Function"); // Runs immediately
            
            await Promise.resolve(); // The "Pause" button
            
            console.log("B: After Await"); // This is now a Microtask!
            }

            console.log("C: Global Start");
            eventLoopTest();
            console.log("D: Global End");
        
        Outout: C -> A -> D -> B 
    
Task Type       Source                                          Priority
Synchronous     "Console logs, variable 
                assignments, loops."                    Highest (Immediate)
Microtasks      "Promise.then, MutationObserver, 
                queueMicrotask."                        High (Runs before next Macrotask)
Macrotasks      "setTimeout, setInterval, 
                setImmediate, I/O, UI Rendering."       Low (One at a time)

RXJS
    RxJS is a library for composing asynchronous and event-based programs by using observable sequences.
    Observable (The Pipe): The source of data.
    Observer (The Tap): The thing that "subscribes" to the data to use it.
    Subscription: User who subscribe and unsubscribe the event.
    Operators (The Filters): Functions like map, filter, or switchMap that transform the data while it's in the pipe.

Feature         Synchronous                 Asynchronous
Execution       Sequential (one by one)     Concurrent (multi-tasking).
Blocking        Blocks the Main Thread      Does not block the main thread.
Performance     Slower for I/O tasks        Faster/Smoother for I/O tasks.
User Experience Can cause UI freezing       Keeps UI responsive.
Handling        Straightforward logic       Uses Callbacks, Promises, or Async/Await.

Feature         .find()                                 QueryBuilder
Complexity      Simple queries only.                    Complex logic, subqueries, and joins.
Performance     Can be slower (fetches all columns)     Faster (you can select only 2 columns).
Readability     High (looks like a configuration)       Moderate (looks like SQL).
Relationships   Uses relations array                    Uses .leftJoin() or .innerJoin().
Best For        Standard CRUD operations                Reports, Dashboards, and complex filters.

switchMap (The "Latest" Map)
    When a new value arrives, it cancels the previous inner Observable and switches to the new one.
    Analogy: A search bar. As you type "A", "AB", "ABC", you only care about the results for "ABC". 
        The previous searches are cancell

mergeMap (The "Parallel" Map)
    It triggers an inner  for every value and lets them all run at the same time. Nothing is cancelled.
    Analogy: A busy restaurant. Multiple orders are placed, and they are served as soon as they are ready, 
        regardless of the order they were placed in.

concatMap (The "Serial" Map)
    It queues the inner Observables. It waits for the current one to finish before starting the next one.
    Analogy: An ATM line. You must finish your transaction before the next person can start theirs.

exhaustMap (The "Ignore" Map)
    It ignores all new values until the current inner Observable has finished.
    Analogy: An elevator button. Once you press it and the elevator is moving, 
        pressing it ten more times does absolutely nothing.

Operator    Strategy    Handling New Values
switchMap   Switch      Cancels the old, starts the new.
mergeMap    Parallel    Runs everything at the same time.
concatMap   Queue       Waits for current to finish, then starts next.
exhaustMap  Ignore      Ignores new values until current is done.

Feature         Primary Key                         Unique Key
Purpose         Uniquely identifies a row           Prevents duplicate values in a column.
Quantity        Only one per table                  Multiple are allowed.
Null Values     Cannot be NULL                      Can have one NULL (usually).
Clustered Index Yes (Default)                       No (Default is Non-Clustered).
Usage           Used as a target for Foreign Keys   Used to prevent duplicates in non-ID fields.

TrackBy
    trackBy is a performance optimization for the *ngFor directive.
    Without trackBy: If you refresh a list of 1,000 items, Angular destroys 
        1,000 DOM elements and creates 1,000 new ones.
    With trackBy: Angular compares the unique IDs. If the IDs are the same, it keeps the existing 
        DOM elements and only updates the content that changed.

Why does Boolean(process.env.MY_FLAG) not work correctly for boolean environment variables?
    Expected answer: Boolean() converts any truthy value to true. Any non-empty string is truthy
    — including the string "false". So Boolean(process.env.MY_FLAG) returns true when 
    MY_FLAG=false is set in the environment, which is the opposite of the intended behaviour. The
    correct idiom is process.env.MY_FLAG === 'true', which returns false for any value other
    than the exact string "true"

What is the difference between passing entity class references and a glob pattern to the

    Expected answer: Class references (
    entities: [User, Post] ) require an explicit import for each
    entity — TypeScript validates the import path at compile time, so a missing entity or mistyped
    path produces a build error. A glob pattern (
    initialize() ,

libuv
    libuv handles everything that happens "outside" the code, like talking to the hard drive, 
    the network, or the computer's operating system.

    Why is it used? (The Main Jobs)
    1. The Event Loop
        This is the heart of Node.js. Libuv manages the "loop" that constantly checks for completed tasks 
        (like a database query coming back) and moves them to the callback queue so JavaScript can execute them.

    2. The Thread Pool
        JavaScript is single-threaded (it can only do one thing at a time). However, some tasks—like reading 
        a file from a disk (fs) or encrypting a password (crypto)—take a long time.
        Libuv maintains a Thread Pool (usually 4 threads by default).
        It offloads these heavy tasks to these background threads so the main JavaScript thread doesn't freeze.

    3. Cross-Platform Consistency
        Operating systems handle networking and files differently (Windows uses IOCP, Linux uses epoll). 
        Libuv acts as a bridge. You write code once in JavaScript, and libuv makes sure it works correctly whether 
        it’s running on a Windows server or a Linux machine.

Pipe Types
    Custom - Defined by ourself ex. reverse a string, calculateTax
    In-Built - which is predefined ex. uppercase, async, lowercase, date
    Pure - 
        A Pure pipe is predictable
        Characteristics:
            Does not change the state of the application.
            Does not call a database or an API.
            Fast and efficient (results can be cached/memoized).
    Impure -
        An Impure pipe is unpredictable or stateful
        Characteristics:
            Might fetch data from a database or external API.
            Depends on global variables or external state.
            Heavier on performance because it usually can't be cached.

Lock Mechanism
    A locking mechanism is a synchronization tool used to prevent multiple users or processes 
    from modifying the same data at the same time. To prevent the Race Condition

providedIn: 'root'
    What it does: Registers the service with the Root Injector.
    Scope: The service is a Singleton (one single instance for the entire app).
    Tree Shaking: ✅ Yes. If your app never actually uses this service, Angular will exclude it from the final production bundle to save space.
    Best for: Most services (API calls, Auth state, Data sharing).

providedIn: 'platform'
    What it does: Registers the service in a special platform-level injector.
    Scope: The service is shared across multiple Angular applications running on the same page.
    Best for: Very rare cases, like a service that manages a shared worker or global window events used by two different micro-frontends on one page.

providedIn: 'any'
    What it does: Creates a unique instance of the service for every eager module, but a shared instance for lazy-loaded modules.
    Scope: Multiple instances might exist if you have multiple top-level modules.
    Best for: Ensuring that lazy-loaded modules get their own "fresh" copy of a service while keeping it tree-shakable.

providedIn: Type<any>
    @Injectable({
        providedIn: UserModule // Only available when UserModule is loaded
    })
    export class UserService { }

If you use 'root', your Admin Panel might see leftover data from the User Profile because they share the same service instance.
If you use 'any', when you load the Admin Panel, Angular creates a totally empty, clean DataCacheService 
just for that module. When you leave the Admin Panel, that specific instance is destroyed.
