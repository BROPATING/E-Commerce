
"Why do you have both src/app.ts and index.ts at the root? Why not just one file?"
    index.ts is the process entry point — its only job is to import the DataSource, initialise the DB connection, 
    then start the HTTP server. app.ts configures the Express application — middleware, routes, error handlers. 
    Separating them means you can import app in a test file without starting a real server or opening a real DB connection. 
    It's the standard pattern used by every production Express codebase.

"Why do you need reflect-metadata and why does import order matter?"
    The bridge that keeps your TypeScript data types alive in the JavaScript runtime.
    TypeScript decorators like @Column() use the Reflect API to attach metadata to class properties at definition time. 
    reflect-metadata is a polyfill that adds this API to the JavaScript runtime. 
    It must be the first import because if any TypeORM entity class is evaluated by the JavaScript engine before the 
    Reflect API exists, the metadata registration calls silently no-op — you get no error, but your entities have no columns, 
    and your DB queries return empty results. 
    It's one of the most common "it works on my machine" bugs in TypeORM projects.

    The "Global Patch" Rule
    reflect-metadata works by "patching" the global Reflect object in JavaScript.
    If you import your Entities or Database Connection before you import reflect-metadata, those entities will try to look for 
    the "notes" (metadata) and find nothing.
    This usually results in a famous error: ColumnTypeUndefinedError: Column type for Product#price is undefined.
    The Correct Sequence
    Think of it like a loading sequence:
        reflect-metadata: Prepare the "note-taking" system.
        dotenv: Load your secret environment variables.
        Entities/Services: Define your logic (now that the note-taking system is ready).
        App/Express: Start the server.
    
"Why ts-node for development but tsc + node dist/ for production?"
    ts-node transpiles TypeScript on the fly in memory — there's no compilation step, so the dev loop is fast. 
    But it adds overhead on every file load and isn't optimised for throughput. In production you pre-compile with tsc to plain 
    JavaScript, which Node runs natively at full speed with no transpilation cost. The build also lets you catch type errors as 
    a CI gate before deployment — ts-node in watch mode is more forgiving about type errors that tsc --strict would reject.


"Why is price_at_purchase on OrderItem instead of just reading product.price at query time?"
    product.price reflects the current price which can be changed by an admin at any time. 
    If we join against it when displaying order history, a customer who paid ₹500 would see ₹750 after a 
    price increase — which is incorrect, potentially a legal issue, and a terrible user experience. 
    By copying the price at the moment the order is created and storing it as a plain decimal column, we make order history 
    immutable to future pricing changes. This is a deliberate, justified denormalisation.

"Why use onDelete: 'RESTRICT' on OrderItem → Product instead of CASCADE?"
    If we used CASCADE, deleting a product from the admin panel would silently delete all OrderItem rows 
    referencing it — destroying historical order data. A customer's past order would become incomplete or disappear. 
    RESTRICT forces the admin to face a database error if they try to delete a product with order history, which surfaces the 
    problem rather than silently corrupting data. The correct solution for the admin would be to mark the product as 
    inactive/hidden rather than hard-deleting it — which is a separate improvement we could add.

"Why is there no separate Cart table — just CartItem directly linked to User?"
    A dedicated Cart entity would only add an intermediate ID with no meaningful attributes of its own. 
    Every customer has exactly one cart at any time, so the CartItem table with a user_id foreign key serves as the implicit cart. 
    This avoids a pointless join and keeps the schema simpler. If requirements ever demanded multiple saved carts per user 
    (wishlists, for example), we'd introduce the Cart entity then — but we don't add tables speculatively.
